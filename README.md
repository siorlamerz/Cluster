# Cluster

Инструмент для удаленного управления серверами через Redis. Агент работает в «стелс-режиме» (в фоне, без вывода в консоль), клиент позволяет рулить всеми машинами из одного места.

Хз буду ли допиливать функционал.

## Структура проекта
```text
.
├── cmd
│   ├── agent      # Сам агент, который живет на серверах
│   ├── builder    # Конфигуратор-сборщик (то, что идет в релизе)
│   └── client     # Панель управления (клиент)
├── internal
│   ├── config     # Логика конфигов и установка systemd сервиса
│   ├── redisclient # Работа с Redis (подписки, команды, стейты)
│   └── remote     # Загрузка конфига с GitHub
├── go.mod
├── go.sum
└── README.md
```

## Как это работает (Кратко!)
1. **Билдер** собирает два бинарника: `agent` и `client`. При сборке он «вшивает» в них настройки (адрес Redis, пароль, имя сервиса) через `-ldflags`.
2. **Агент** ставится на сервер. Если запущен от root, он сам создает systemd-сервис, копирует себя в указанную папку и уходит в фон. Стучится в Redis раз в X секунд (heartbeat).
3. **Клиент** подключается к тому же Redis, видит список всех живых агентов и может рассылать им команды.

## Быстрый старт через Билдер
В релизе поставляется только `builder`. Это единственный файл, который тебе нужен для подготовки системы.

1. Запускаешь билдер: `./builder`
2. Заполняешь поля:
   - **Redis Address**: IP и порт твоего Redis сервера.
   - **Redis Password**: Пароль от Redis.
   - **Github Config URL**: Ссылка на `.json` с конфигом (если хочешь обновлять настройки удаленно).
   - **Service Name**: Имя, под которым агент будет висеть в `systemctl` (например, `sys-monitor`).
   - **Remote Install Path**: Путь, куда агент скопирует себя на сервере (например, `/usr/local/bin/sys-monitor`).
   - **Heartbeat Interval**: Как часто агент говорит в редис что жив (например, `30s`).
   - **Client Response Timeout**: Сколько клиент ждет ответа от сервера (например, `10s`).
   - **Local Build Output Dir**: Папка, куда забилдятся готовые бинарники (по дефолту `./build`).

3. Жмешь **Enter**. В папке `build` получаешь два файла: `agent` и `client`.

## Сборка из исходников (без билдера)
Можно (но зачем) собрать всё руками через консоль, используй флаг `-ldflags`. Это единственный способ прокинуть настройки в бинарник без правки кода.

**Пример команды для сборки агента:**
```bash
go build -ldflags "-X 'cluster/internal/config.DefaultRedisAddr=1.2.3.4:6379' \
-X 'cluster/internal/config.DefaultRedisPass=your_pass' \
-X 'cluster/internal/config.ServiceName=my_agent' \
-X 'cluster/internal/config.BinaryInstallPath=/usr/local/bin/my_agent' \
-X 'cluster/internal/config.HeartbeatInterval=30s'" \
-o agent ./cmd/agent/main.go
```

**Для клиента:**
```bash
go build -ldflags "-X 'cluster/internal/config.DefaultRedisAddr=1.2.3.4:6379' \
-X 'cluster/internal/config.DefaultRedisPass=your_pass' \
-X 'cluster/internal/config.ResponseTimeout=10s'" \
-o client ./cmd/client/main.go
```

## Использование клиента
Запускаешь `./client` и используешь команды:

- `list` — показать всех агентов, их статус (Online/Offline) и время последнего отклика.
- `exec <device_id> <command>` — выполнить команду на конкретной машине.
- `exec-all <command>` — разослать команду всем серверам сразу.
- `set-github <device_id> <url>` — сменить URL конфига для конкретного агента.
- `prune` — вычистить из списка Redis все мертвые устройства.
- `uninstall-all` — **ЯДЕРНАЯ КНОПКА**. Полностью удаляет агента, сервис и все файлы со всех машин. Требует двойного подтверждения.
- `help` — список всех команд.

## Redis
Проект использует Redis как шину данных. 

- **`devices_state`** (Hash): Хранит JSON со стейтом каждого устройства (ID, Hostname, IsRoot и т.д.).
- **`heartbeat:<device_id>`** (String): Ключ с TTL. Если ключ есть — агент Online.
- **`commands:<device_id>`** (Pub/Sub): Канал для команд конкретному агенту.
- **`commands:all`** (Pub/Sub): Общий канал для рассылки всем.
- **`results:<device_id>`** (Pub/Sub): Канал, по которому агент возвращает вывод команды клиенту.

## Важно
- **Root-права**: Чтобы агент установил себя как сервис и мог выполнять `sudo` команды, его нужно первый раз запустить от root: `sudo ./agent`.
- **Стелс-режим**: Агент ничего не пишет в консоль. Если нужно проверить, что с ним, используй:
  `journalctl -u <имя_сервиса> -f`
